# Development Spec

## 文档目的

用于指导 `PathoMind 医疗知识与质控助手` 的需求拆解、数据准备、研发实现、测试验收与发布演示。

## 与其他 spec 的关系

- 产品定义以 `PRODUCT_BRIEF.md` 为准
- 功能范围与验收以 `PRD.md` 为准
- 数据准备以 `DEMO_DATA_CHECKLIST.md` 为准
- 演示与回归以 `DEMO_SCENARIOS.md` 为准
- agent 执行方式以 `AGENT_RUNBOOK.md` 为准

## 开发总原则

### 业务优先于炫技

任何功能必须先回答：

1. 是否解决目标用户问题
2. 面试官是否能快速听懂
3. 是否能稳定演示

### 可信优先于花哨

- 回答可追溯
- 数据来源清晰
- 输出边界明确
- 评估过程可解释

### 先闭环再复杂化

开发顺序固定为：

1. 跑通最小闭环
2. 补强场景能力
3. 再做体验优化

## 范围

### 第一阶段必须完成

- 医疗化命名与 README 重构
- demo 数据准备与知识库构建
- 检索 + 引用回答闭环
- 关键页面医疗化文案改造
- 标准演示题集
- 基础评估结果

### 暂不纳入

- 自动诊断
- 医院生产环境部署
- 多租户权限系统
- 患者敏感数据闭环

## 项目阶段路线

### 第 1 步：Spec / Skill 基础层

- 目标：统一真源、执行协议、状态回写、求职展示层
- 当前状态：已完成

### 第 2 步：Demo 数据层

- 目标：落一版最小可用公开医疗数据包
- 交付重点：
  - 2 到 3 份指南 / 规范
  - 3 到 5 份 SOP / 流程文档
  - 1 到 2 份设备说明书
  - 1 份培训材料
- 验收重点：
  - 来源可说明
  - 命名规范一致
  - 无隐私字段
  - 每份资料可映射到至少一个演示场景

### 第 3 步：问答闭环层

- 目标：让“导入 -> 检索 -> 引用回答 -> 边界拒答”真实跑通
- 交付重点：
  - SOP 查询
  - 指南查询
  - 术语解释
  - 设备操作
  - 图文检索
  - 边界拒答
- 验收重点：
  - 固定题集中大多数题目返回带引用结果
  - 边界问题不输出诊断类结论

### 第 4 步：展示界面层

- 目标：让 Dashboard 首屏呈现医疗产品，而不是通用 RAG 后台
- 交付重点：
  - 医疗化标题
  - 医疗化页面命名
  - 示例问题与演示说明
  - 首页核心场景说明
- 验收重点：
  - 面试官 3 分钟内能听懂定位、能力、边界

### 第 5 步：评估与回归层

- 目标：把 demo 从“一次能跑”升级成“稳定能演示”
- 交付重点：
  - 固定题集
  - 最小评估结果
  - 回归记录
- 验收重点：
  - 每次重要改动后可复跑
  - 回归失败可记录和定位

### 第 6 步：求职包装层

- 目标：把结果同步到简历、讲稿、展示页和面试话术
- 交付重点：
  - 项目描述
  - 1 分钟讲稿
  - 演示截图
  - 面试回答模板
- 验收重点：
  - 1 分钟讲定位
  - 3 分钟讲闭环
  - 5 分钟讲技术深度

## 当前阶段定义

当前阶段定义为：`文档与协作层完成，进入 MVP 演示实现层`

当前优先顺序固定为：

1. demo 数据方案
2. 标准演示题集
3. Dashboard 医疗化文案
4. 演示回归验证
5. MCP 对外文案医疗化
6. 简历与面试材料同步

## 数据规范

- 只允许公开、可说明来源的资料进入 demo
- 禁止使用患者姓名、检查号、住院号、联系方式、可识别隐私图像
- 文件命名采用：`[type]_[topic]_[version_or_year].ext`

## 检索与回答规范

默认链路：

1. Query 预处理
2. Dense Retrieval
3. Sparse Retrieval
4. RRF 融合
5. Rerank
6. 引用式回答组装

回答必须尽量具备：

- 简短结论
- 关键依据
- 来源引用
- 限制说明

## Dashboard 规范

页面目标是“面试展示”，不是“后台炫技”。

推荐页面：

- 科室知识概览
- 医疗资料导入中心
- 医疗资料浏览
- 问答链路与引用分析
- 评估面板

## 测试与评估

每次重要迭代至少验证：

1. 文档可以成功导入
2. 检索结果能返回相关片段
3. 引用信息完整
4. 不相关问题不胡乱编造
5. 页面能展示关键结果

建议指标：

- Hit Rate@K
- 引用完整率
- 回答相关性
- 幻觉率
- 边界问题拒答质量

## 发布与演示规范

每次发布前必须检查：

- README 与定位一致
- demo 数据可正常导入
- 标准演示问题稳定通过
- 页面截图更新
- 简历口径同步

## Definition of Done

一个功能或阶段完成，至少满足：

- 业务目标明确
- 文档已同步
- 基础测试通过
- 可稳定演示
- 输出边界清晰
- 对求职叙事有明确增益
